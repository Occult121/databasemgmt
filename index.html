check -pt method of log recovery


Where a database is to be recovered from hd crash we use check pt method of log recovery .
a check pt is a special log record , the main stages in this recovery using check –pts as :
Output onto stable storage all log records residing In main memory
Output all modified buffer blocks onto disk
output onto stable storage a log record check –pt
No database modifications can take place where a check –pt in progress

we now suppose that a hd crash occurs where a transaction executing , the log is consulted backwds
till finds the transaction –commit , log is still
consulted backwds till finds the check –pt , log is still consulted backwds till finds transaction – start , that identifies the transaction.
Now , undo /redo operations can be performed on this transaction and any others . for concurrent transactions processing , fuzzy check –pt is used



Acid properties of transaction


A transaction is a unit of program execution that accesses and –or possibly updates various data items . Acid properties of a transaction are :
Atomicity --- either transaction executes in entirety or none executes at all
Consistency ---- state of database remains same both before & after transaction , despite system changes
Isolation --- despite concurrent execution of transactions , each transaction proceeds independently of others , despite system changes
Durability ----- any modifications made to database as a result of transaction , persist even if there are system changes

e.gs there is a schedule involving 2 accts a and b, initial values in a and b accts are $ 1000 and $ 2000 .
a transaction occurs that transfers $ 50 from a to b , as a result of which , final values in a and b accts become $950 and $ 2050; clearly
a consistent state as sum a +b is preserved both before & after the transaction . we now
suppose that a hd crash occurs after write – a committed but before write –b committed , so ,
now , final values in a and b becomes $ 950 and $ 2000 , clearly an inconsistent state , as sum a +b is no more
preserved both before and after the transaction.even if system failure did not occurs and transaction proceeded to completion,
this inconsistent state would not have been visible except during occurrence of transaction.
This again reflects atomicity , isolation gets preserved , durability gets preserved through recovery mgmt. component of transactions
